## 1. 수용 기준 개요

### 1.1 정의 완료 (Definition of Done)

Port Chaser 기능은 다음 조건이 모두 충족될 때 완료로 간주합니다:

1. 모든 EARS 수용 기준이 충족됨
2. 모든 테스트 시나리오가 통과함
3. 코드 커버리지가 85% 이상임
4. LSP Quality Gates가 통과함 (0 errors, 0 type errors, 0 lint errors)
5. 모든 엣지 케이스가 처리됨
6. 성능 목표가 충족됨

### 1.2 품질 게이트 (Quality Gates)

| 게이트 | 기준 | 측정 방법 |
|--------|------|----------|
| 테스트 커버리지 | >= 85% | `go test -cover` |
| 단위 테스트 | 모두 통과 | `go test ./...` |
| LSP Errors | 0 | `gopls check` |
| LSP Type Errors | 0 | `gopls check` |
| LSP Lint Errors | 0 | `golangci-lint run` |
| 시작 시간 | < 2초 | 수동 측정 |
| 스캔 시간 | < 500ms | 벤치마크 |

## 2. 사용자 스토리별 수용 기준

### US-001: 포트 사용 현황 목록 표시

#### AC-PORT-001: 2초 이내 시작

**Given** 사용자가 터미널에서 Port Chaser를 실행할 때
**When** 애플리케이션이 시작되면
**Then** 시스템은 2초 이내에 포트 목록을 표시해야 한다

**테스트 방법**:
```bash
time ./port-chaser
# 실행 시간이 2초 미만이어야 함
```

#### AC-PORT-002: 필수 정보 표시

**Given** 포트 목록이 로드되었을 때
**When** 사용자가 목록을 확인하면
**Then** 각 항목은 포트 번호, 프로세스 이름, PID를 포함해야 한다

**테스트 방법**:
- TUI 출력에서 각 포트 항목 확인
- 포트 번호(정수) 존재 확인
- 프로세스 이름(문자열) 존재 확인
- PID(정수) 존재 확인

#### AC-PORT-003: 일반 포트 우선 표시

**Given** 여러 포트가 사용 중일 때
**When** 포트 목록이 표시되면
**Then** HTTP(80, 8080), HTTPS(443), 3000, 5000 포트가 상단에 표시되어야 한다

**테스트 방법**:
- 일반 포트(3000, 8080)와 다른 포트(9999)가 모두 실행 중인 상황에서 목록 확인
- 일반 포트가 상단에 위치하는지 확인

### US-002: TUI 네비게이션

#### AC-TUI-001: 방향키 하이라이트

**Given** 포트 목록이 표시되어 있을 때
**When** 사용자가 방향키(위/아래)를 입력하면
**Then** 현재 선택 항목이 하이라이트되어야 한다

**테스트 시나리오**:
```gherkin
Scenario: 방향키로 항목 선택
  Given 포트 목록에 5개의 항목이 있고
    And 첫 번째 항목이 선택되어 있을 때
  When 사용자가 아래쪽 방향키를 누르면
  Then 두 번째 항목이 선택되어야 하고
    And 선택 항목이 하이라이트되어야 한다

Scenario: 위쪽 방향키
  Given 두 번째 항목이 선택되어 있을 때
  When 사용자가 위쪽 방향키를 누르면
  Then 첫 번째 항목이 선택되어야 한다
```

#### AC-TUI-002: Enter 키 종료

**Given** 포트 항목이 선택되어 있을 때
**When** 사용자가 Enter 키를 누르면
**Then** 시스템은 선택한 프로세스에 SIGTERM을 전송해야 한다

**테스트 시나리오**:
```gherkin
Scenario: Enter 키로 프로세스 종료
  Given 포트 3000에서 "test-app"이 실행 중이고
    And 해당 항목이 선택되어 있을 때
  When 사용자가 Enter 키를 누르면
  Then SIGTERM 신호가 프로세스에 전송되어야 하고
    And 확인 다이얼로그가 표시되어야 한다
```

#### AC-TUI-003: SIGKILL 폴백

**Given** 프로세스가 SIGTERM을 수신했지만 종료하지 않았을 때
**When** 3초가 경과하면
**Then** 시스템은 SIGKILL을 전송해야 한다

**테스트 시나리오**:
```gherkin
Scenario: SIGTERM 무시 프로세스 강제 종료
  Given SIGTERM을 무시하는 테스트 프로세스가 실행 중이고
    And 사용자가 Enter 키를 눌러 종료 요청했을 때
  When 3초가 경과하면
  Then 시스템은 SIGKILL을 전송해야 하고
    And 프로세스가 종료되어야 한다
```

#### AC-TUI-004: q 키 종료

**Given** TUI가 실행 중일 때
**When** 사용자가 q 키를 누르면
**Then** TUI가 종료되어야 한다

**테스트 방법**:
- TUI 실행 후 'q' 키 입력
- 프로세스가 정상 종료하는지 확인

Scenario: 종료 이력 자동 기록

Scenario: 히스토리 뷰 표시

Scenario: 데이터베이스 잠금 처리

## 3. 엣지 케이스 수용 기준

### EC-001: 권한 문제

**Given** 사용자가 시스템 중요 프로세스를 종료하려 할 때
**When** 종료 요청이 있으면
**Then** 경고 다이얼로그를 표시해야 한다

**테스트 시나리오**:
```gherkin
Scenario: 시스템 프로세스 보호
  Given PID 1(init) 프로세스가 선택되어 있을 때
  When 사용자가 Enter 키를 누르면
  Then 경고 다이얼로그가 표시되어야 하고
    And "정말 종료하시겠습니까?" 메시지가 표시되어야 한다
```

### EC-002: Docker 데몬 연결 실패

**Given** Docker 데몬이 실행 중이지 않을 때
**When** 애플리케이션이 시작되면
**Then** Docker 기능을 자동 비활성화하고 일반 포트만 표시해야 한다

### EC-003: 빈 포트 목록

**Given** 활성 포트가 없을 때
**When** 애플리케이션이 시작되면
**Then** "활성 포트가 없습니다" 메시지를 표시해야 한다

### EC-004: 프로세스 종료 경쟁 조건

**Given** 종료 신호 전송 후 프로세스가 이미 종료되었을 때
**When** SIGKILL을 전송하려 하면
**Then** "이미 종료된 프로세스입니다" 메시지를 표시해야 한다

### EC-005: 데이터베이스 잠금

**Given** SQLite 데이터베이스가 잠겨 있을 때
**When** 쓰기를 시도하면
**Then** 로깅만 수행하고 TUI를 중단하지 않아야 한다

### EC-006: 비표준 포트 범위

**Given** 0-1023 범위의 시스템 포트가 사용 중일 때
**When** 포트 목록이 표시되면
**Then** 모든 포트 범위를 지원하고 시스템 포트를 시각적으로 구분해야 한다

### EC-007: Windows 호환성

**Given** Windows 운영체제에서 실행될 때
**When** TUI가 렌더링되면
**Then** 플랫폼별 차이를 추상화하고 정상 작동해야 한다

## 4. 성능 수용 기준

**Given** 사용자가 Port Chaser를 실행할 때
**When** 애플리케이션이 초기화되면

```bash
hyperfine './port-chaser'
# 평균 실행 시간이 2초 미만이어야 함
```

### PERF-002: 스캔 시간 < 500ms

**Given** 포트 스캔이 요청되었을 때
**When** 스캔이 완료되면
**Then** 500ms 이내에 결과를 반환해야 한다

**테스트 방법**:
```go
benchmarks := []struct{
    name string
    f func(b *testing.B)
}{
    {"Scan", benchmarkScan},
}
```

### PERF-003: 검색 응답 < 100ms

**Given** 사용자가 검색어를 입력할 때
